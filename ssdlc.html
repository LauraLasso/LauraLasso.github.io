F4
Implementación Segura
Durante el desarrollo · OWASP Secure Coding Practices
Validación de entradas
Validar siempre en el servidor, nunca solo en el cliente
Usar listas blancas (qué se permite), no listas negras
Rechazar cualquier input que no cumpla el formato esperado
Validar tipo, rango y longitud de todos los datos de entrada
Identificar todas las fuentes de datos y clasificarlas como confiables o no confiables
Usar una rutina centralizada de validación para toda la aplicación
Codificación de salidas
Codificar contextualmente toda salida de datos que provenga de fuentes no confiables
Codificar salidas para HTML, JavaScript, CSS, URL y SQL según el contexto
Nunca insertar datos no confiables directamente en el DOM sin sanitizar
Sanitizar toda salida de datos no confiables en consultas SQL, XML y LDAP
Autenticación
Contraseñas hasheadas con bcrypt o Argon2
2FA para accesos sensibles
Bloqueo tras intentos fallidos (rate limiting)
Recuperación de contraseña con tokens de un solo uso y expiración
Requerir autenticación para todas las páginas y recursos no públicos
Usar servicios de autenticación estándar probados, nunca implementación propia
Mensajes de error genéricos — nunca revelar si el usuario existe o no
Gestión de sesiones
Tokens con expiración corta
Cookies con flags HttpOnly y Secure
Invalidar sesión al cerrar sesión de verdad
Generar un nuevo identificador de sesión tras el login (previene session fixation)
No transmitir identificadores de sesión en la URL
Establecer un tiempo de inactividad máximo que cierre la sesión automáticamente
Control de acceso (Autorización)
Control de acceso basado en roles (RBAC)
Verificar permisos en cada petición, no solo al login
Nunca tomar decisiones de autorización en el cliente
Principio de mínimo privilegio — conceder solo los permisos estrictamente necesarios
Denegar el acceso por defecto; requerir concesión explícita
Registrar todos los fallos de control de acceso y alertar cuando sea apropiado
Criptografía
TLS 1.2 mínimo (preferible 1.3) para toda comunicación
AES-256 para datos en reposo
Nunca implementar algoritmos criptográficos propios
Nunca usar algoritmos obsoletos: MD5, SHA-1, DES
Generar números aleatorios con módulos aprobados — nunca Math.random() para seguridad
Proteger las claves maestras del acceso no autorizado; rotarlas periódicamente
Manejo de errores y logs
Nunca mostrar stack traces ni mensajes técnicos al usuario
Loggear errores internamente con detalle; mensajes genéricos al exterior
Registrar todos los eventos de seguridad: logins, fallos de autenticación, cambios de privilegios
Proteger los logs contra modificación o acceso no autorizado
No registrar datos sensibles en los logs: contraseñas, tokens, datos personales
Protección de datos
Eliminar datos sensibles de la caché del navegador (Cache-Control: no-store)
No almacenar contraseñas, claves de conexión u otros datos sensibles en el código fuente
Deshabilitar el autocompletado del navegador en formularios con datos sensibles
Aplicar el principio de mínima exposición: no recoger datos que no sean necesarios
Seguridad en comunicaciones
Usar TLS para toda comunicación que transmita datos sensibles
Validar los certificados TLS del servidor en conexiones externas
Usar HSTS (Strict-Transport-Security) para forzar HTTPS en el navegador
No mezclar contenido HTTP y HTTPS en la misma página (mixed content)
Configuración del sistema
Mantener los componentes del servidor actualizados con los últimos parches de seguridad
Eliminar todas las funcionalidades, archivos y código de prueba antes de producción
Deshabilitar los métodos HTTP innecesarios (PUT, DELETE, TRACE si no se usan)
Configurar el servidor para no revelar versiones de software en cabeceras HTTP
Restringir permisos del proceso del servidor al mínimo necesario
Seguridad de base de datos
Usar prepared statements o procedimientos almacenados para todas las consultas
El usuario de BD de la app debe tener solo los permisos mínimos necesarios
Cerrar las conexiones a BD tan pronto como dejen de ser necesarias
Nunca conectarse a la BD con credenciales de administrador desde la aplicación
Cifrar la comunicación entre aplicación y base de datos
Gestión de archivos
Validar el tipo real del fichero subido — no confiar solo en la extensión o el MIME declarado
Nunca ejecutar ficheros subidos por el usuario en el servidor
Almacenar los ficheros subidos fuera del directorio raíz del servidor web
Limitar el tamaño máximo de los ficheros subidos
Renombrar los ficheros subidos para evitar path traversal o conflictos
Gestión de memoria y código
Liberar los recursos (conexiones, ficheros, memoria) tan pronto como dejen de usarse
Usar mecanismos de sincronización para evitar condiciones de carrera en recursos compartidos
Verificar la integridad de librerías y ejecutables con checksums o hashes
Usar solo librerías y componentes de terceros de fuentes confiables y mantenidas activamente
Seguridad continua en el CI — Shift Left

La seguridad no espera a F5. En cada commit se ejecutan automáticamente las siguientes comprobaciones, detectando errores mientras se escribe el código — cuando son 100x más baratos de corregir que en producción.

Tipo de comprobaciónHerramientaCuándo se ejecuta
Análisis estático (SAST) Bandit · SpotBugs · SonarQube · ConQAT · Parasoft En cada commit — detecta código inseguro, patrones de vulnerabilidad y deuda técnica de seguridad sin ejecutar la app
Análisis de dependencias OWASP Dependency Check En cada PR — detecta librerías de terceros con CVEs conocidos antes de que entren en la rama principal
Secretos en código Gitleaks · CredScan En cada commit — detecta credenciales, tokens o claves API hardcodeadas accidentalmente en el repositorio

Verificación y Pruebas
Antes de desplegar · OWASP Testing Guide (WSTG)

Con la app completa desplegada en staging, se verifica que el conjunto funciona de forma segura bajo ataque real. SAST y dependencias ya se revisaron en F4 — aquí se añade todo lo que requiere la app en ejecución.

Tipo de pruebaHerramientaQué detecta
Análisis dinámico (DAST) OWASP ZAP · Burp Suite · Acunetix · Arachni · Wfuzz · Microsoft AppVerifier Vulnerabilidades en la app en ejecución — XSS, SQLi básico, headers inseguros, CSRF, SSRF, errores de configuración HTTP
Análisis interactivo (IAST) Contrast Security · Seeker (Synopsys) · Hdiv Security Vulnerabilidades en runtime con trazabilidad exacta hasta la línea de código — flujo de datos contaminado (taint analysis) detectado desde dentro de la app mientras se ejecutan las pruebas DAST
Pruebas de penetración nmap · ffuf · SQLmap · Metasploit · jwt_tool · Manual (WSTG) Explotación real de vulnerabilidades siguiendo la metodología OWASP WSTG: nmap — reconocimiento: puertos abiertos, versiones de servicios expuestos
ffuf — enumeración: endpoints y directorios ocultos no enlazados en el HTML
SQLmap — confirmar y explotar inyecciones SQL (blind, time-based, error-based)
jwt_tool — probar alg:none, firma débil y expiración incorrecta en tokens JWT
Metasploit — post-explotación: medir alcance real del daño tras una brecha
Manual (WSTG) — IDOR, lógica de negocio rota, escalada de privilegios
Revisión con ASVS Checklist manual Verificar cumplimiento de requisitos definidos en F2 — lo que las herramientas automáticas no ven: credenciales hardcodeadas, cifrado ausente en BD, logs no configurados

Compliance — Normativas y Regulaciones
Transversal a todo el SDLC · Se define en F1 y se verifica en F5

El cumplimiento normativo no es una fase aislada — se identifica en F1 (qué leyes aplican al proyecto), se traduce en requisitos técnicos en F2, se implementa en F4 y se verifica formalmente antes del despliegue en F5. El profesional responsable es el DPO (Data Protection Officer) o Compliance Officer.

Código Seguro por Diseño

Principios de diseño que hacen que los errores de seguridad sean imposibles de cometer, no solo difíciles. Ejemplos en Java.

Bloque 0 Fundamentos del Diseño Seguro
P1 La ambigüedad como origen del riesgo
❌ Mal — constructor ambiguo, el orden importa y es confuso
User user = new User(
  "user@example.com",
  "ADMIN"
);
✅ Bien — setters explícitos que dejan claro qué es cada cosa
User user = new User();
user.setEmail("user@example.com");
user.setRole("ADMIN");
P2 Aprovechar el sistema de tipos
❌ Mal — id y role son Strings genéricos, el compilador no puede ayudar
class User {
    String id;
    String role;
}
✅ Bien — tipos específicos del dominio, el compilador detecta usos incorrectos
class UserId { String value; }
class Role   { String value; }
class User   {
    UserId id;
    Role role;
}
P3 Diseño como primera línea de defensa
❌ Mal — comprobaciones manuales encadenadas, frágil
void deleteUser(User req, User target) {
    if (req.isActive()) {
        if (req.isAdmin()) {
            // eliminar usuario
        }
    }
}
✅ Bien — AdminUser hace imposible construir un usuario no válido
final class AdminUser {
    final User user;
    AdminUser(User user) {
        if (!user.active)
            throw new IllegalStateException();
        this.user = user;
    }
}
void deleteUser(AdminUser req, User target) {
    // Sin más comprobaciones necesarias
}
P4 Diseño defensivo frente a diseño seguro
❌ Mal — validaciones dispersas en cada método que usa amount
void withdraw(Account account, int amount) {
    if (amount <= 0)
        throw new IllegalArgumentException();
    if (account == null)
        throw new IllegalStateException();
    if (account.getBalance() < amount)
        throw new IllegalStateException();
    account.decrease(amount);
}
✅ Bien — Money no puede existir si es inválido
final class Money {
    final int value;
    Money(int value) {
        if (value <= 0)
            throw new IllegalArgumentException();
        this.value = value;
    }
}
void withdraw(Account account, Money amount) {
    account.decrease(amount);
}
P5 No representar lo inválido
❌ Mal — Email es un String, puede contener cualquier cosa
class Email {
    String value;
}
✅ Bien — si el email no es válido, el objeto no se puede construir
final class Email {
    private final String value;
    Email(String value) {
        if (value == null ||
            !value.matches(
                "^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$"))
            throw new IllegalArgumentException(
                "Invalid email");
        this.value = value;
    }
}
P6 Validar tarde es asumir riesgo
❌ Mal — el email viaja como String y se valida al final
class User { String email; }

void process(User user) {
    sendEmail(user.email);
}
void sendEmail(String email) {
    if (!email.matches(EMAIL_REGEX))
        throw new IllegalArgumentException();
}
✅ Bien — se valida en construcción; si el objeto existe, ya es válido
class User { Email email; }

final class Email {
    final String value;
    Email(String value) {
        if (!value.matches(EMAIL_REGEX))
            throw new IllegalArgumentException();
        this.value = value;
    }
}
P7 Secure by Design frente a Secure by Policy
❌ Mal — la política existe en un documento, pero el código la ignora
// POLÍTICA: Validar email antes de operar.
class User {
    String email;
    boolean emailVerified;
}
void performAction(User user) {
    // Se asume que alguien verificó...
    doSomething();
}
✅ Bien — VerifiedUser hace imposible operar sin haber verificado
final class VerifiedUser {
    final String email;
    VerifiedUser(String email) {
        this.email = email;
    }
}
void performAction(VerifiedUser user) {
    doSomething();
}
P8 El problema de la confianza implícita
❌ Mal — userId viene de HTTP pero se trata como de confianza
class RequestContext { String userId; }

void process(RequestContext ctx) {
    // se asume fiable, pero viene del exterior
    loadUserData(ctx.userId);
}
✅ Bien — ExternalUserId vs AuthenticatedUserId distinguen el origen
final class ExternalUserId {
    final String value;
    ExternalUserId(String value) {
        if (!value.matches("\\d+"))
            throw new IllegalArgumentException();
        this.value = value;
    }
}
final class AuthenticatedUserId {
    final String value;
    // Solo se construye tras autenticar
    AuthenticatedUserId(ExternalUserId id) {
        this.value = id.value;
    }
}
void process(AuthenticatedUserId userId) {
    loadUserData(userId.value);
}
P9 Separación clara de responsabilidades
❌ Mal — UserService mezcla validación, normalización y lógica
class UserService {
    void createUser(String email) {
        if (email == null || !email.contains("@"))
            throw new IllegalArgumentException();
        String normalized = email.toLowerCase();
        saveToDatabase(normalized);
        sendWelcomeEmail(normalized);
    }
}
✅ Bien — Email encapsula validación; el servicio solo orquesta
final class Email {
    final String value;
    Email(String value) {
        if (value == null || !value.contains("@"))
            throw new IllegalArgumentException();
        this.value = value.toLowerCase();
    }
}
class UserService {
    void createUser(Email email) {
        saveToDatabase(email.value);
        sendWelcomeEmail(email.value);
    }
}
P10 Evitar la lógica de seguridad dispersa
❌ Mal — el mismo if (!user.isAuthenticated()) repetido en cada método
void updateProfile(User user, String email) {
    if (!user.isAuthenticated())
        throw new SecurityException();
    save(email);
}
void sendNewsletter(User user) {
    if (!user.isAuthenticated())
        throw new SecurityException();
}
void deleteAccount(User user) {
    if (!user.isAuthenticated())
        throw new SecurityException();
}
✅ Bien — AuthenticatedUser centraliza la comprobación una sola vez
final class AuthenticatedUser {
    final User user;
    AuthenticatedUser(User user) {
        if (!user.isAuthenticated())
            throw new SecurityException();
        this.user = user;
    }
}
void updateProfile(AuthenticatedUser u, Email email) {
    save(email.value());
}
void sendNewsletter(AuthenticatedUser u) { ... }
void deleteAccount(AuthenticatedUser u) { ... }
P11 Diseñar pensando en el error humano y la complejidad
❌ Mal — árbol de if anidados, fácil olvidar una condición
if (user != null) {
  if (user.isActive()) {
    if (user.hasRole("ADMIN") ||
        user.hasRole("SUPPORT")) {
      if (!user.isLocked()) {
        if (request.isFromInternalNetwork()) {
          performSensitiveAction();
        }
      }
    }
  }
}
✅ Bien — AuthorizedAdmin agrupa todas las condiciones
class AuthorizedAdmin {
    private final User user;
    AuthorizedAdmin(User user) {
        if (!user.isActive() ||
            user.isLocked() ||
            !user.hasRole("ADMIN"))
            throw new SecurityException();
        this.user = user;
    }
    void performSensitiveAction() { ... }
}

new AuthorizedAdmin(currentUser)
    .performSensitiveAction();
Bloque 2 Domain Primitives
DP1 Confusión de parámetros — el compilador como guardián
❌ Mal — tres Strings, el compilador no detecta el orden incorrecto
class AuthService {
    boolean authenticate(
        String userId,
        String sessionToken,
        String password) { ... }
}
// ERROR GRAVÍSIMO pero compila sin quejarse
auth.authenticate(password, userId, sessionToken);
✅ Bien — tipos distintos hacen imposible confundir el orden
final class UserId       { ... }
final class SessionToken { ... }
final class Password     { ... }

class AuthService {
    boolean authenticate(
        UserId userId,
        SessionToken token,
        Password password) { ... }
}
// Esto YA NO COMPILA
auth.authenticate(password, userId, token);

El compilador detecta el error en tiempo de compilación, no en producción.

DP2 Domain Primitives y ataques de inyección
❌ Mal — String id acepta cualquier payload malicioso
// Acepta: "1 OR 1=1", "../../../etc/passwd"
// Cualquier String se convierte en ID
void getTeacher(String teacherId) { ... }
✅ Bien — TeacherId solo acepta enteros positivos
public final class TeacherId {
    private final int value;
    private TeacherId(int value) {
        if (value <= 0)
            throw new IllegalArgumentException(
                "TeacherId must be a positive integer");
        this.value = value;
    }
    public static TeacherId of(int value) {
        return new TeacherId(value);
    }
    public int value() { return value; }
}

Cualquier payload de SQL Injection o path traversal es imposible de construir: el constructor lo rechaza.

DP3 Eliminación de ambigüedad semántica
❌ Mal — String id no comunica nada, cualquier ID es válido
public void process(String id) { ... }
process(order.getOrderId());   // compila
process(user.getUserId());     // también compila — ¡error silencioso!
✅ Bien — el tipo hace que solo el ID correcto compile
public void process(TeacherId id) { ... }
process(user.getUserId());      // Error de compilación
process(teacher.getTeacherId()); // Correcto
DP4 Inmutabilidad como refuerzo de seguridad
❌ Mal — mutable, cualquiera puede cambiar el valor después
class TeacherId {
    private int value;
    public void setValue(int value) {
        this.value = value;  // ← peligro
    }
}
✅ Bien — inmutable y constructor privado; el valor no puede cambiar
public final class TeacherId {
    private final int value;      // final = inmutable
    private TeacherId(int value) { // private = factory method
        if (value <= 0)
            throw new IllegalArgumentException();
        this.value = value;
    }
    public static TeacherId of(int value) {
        return new TeacherId(value);
    }
    public int value() { return value; }
}
DP5 Errores como estados imposibles
❌ Mal — hay que validar null y blank en cada método
public void assign(String teacherId) {
    if (teacherId == null || teacherId.isBlank())
        throw new IllegalArgumentException();
    ...
}
✅ Bien — si TeacherId existe, ya es válido; sin comprobaciones extra
public void assign(TeacherId teacherId) {
    // Si llegamos aquí, teacherId ya fue validado
    // en construcción. No hay nada más que comprobar.
    ...
}
DP6 Domain Primitives y testing de seguridad
❌ Mal — tests de validación dispersos en cada clase que usa el tipo
// Hay que repetir el test en cada clase
// que reciba el parámetro como String
@Test
void withdrawRejectsNegativeAmount() {
    assertThrows(..., () -> withdraw(account, -10));
}
@Test
void assignRejectsNullId() {
    assertThrows(..., () -> assign(null));
}
✅ Bien — las invariantes se prueban directamente sobre el tipo, en un solo lugar
final class Balance {
    private final BigDecimal amount;
    private Balance(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException(
                "Negative balance not allowed");
        this.amount = amount;
    }
    static Balance of(BigDecimal amount) {
        return new Balance(amount);
    }
}

@Test
void balanceCannotBeNegative() {
    assertThrows(IllegalArgumentException.class,
        () -> Balance.of(new BigDecimal("-10"))
    );
}

Un solo test unitario sobre el Domain Primitive cubre todas las clases que lo usan.

CI/CD con GitHub Actions

Dos workflows completos siguiendo el ciclo SDL/SDLC: el CI corre en cada commit y PR, el CD gestiona el despliegue a staging y producción con aprobación manual (equivalente a la FSR). Este CI/CD sólo automatizan las 3 últimas fases del SDLC, que son la implementación (CI), la verificación (CI/CD) y el despliegue (CD).

Modelo de branches y gates de seguridad
feature/* ──→ Pull Request ──→ 🔍 CI Security ──→ develop
✕ Falla → merge bloqueado
✓ Pasa → merge permitido
Gitleaks CodeQL Dependency-Check BinSkim
develop ──→ Pull Request ──→ 🔍 CI Security + DAST ──→ main
✕ Falla → merge bloqueado
✓ Pasa → merge permitido
Gitleaks CodeQL Dependency-Check BinSkim + OWASP ZAP
main ──→ Deploy staging ──→ 🚀 CD Staging ──→ ⏳ Aprobación manual (FSR) ──→ producción
✕ Rechazado → no se despliega
✓ Aprobado → deploy a producción

Aprobación manual — equivalente a la FSR
SDL: Final Security Review · un revisor autorizado debe aprobar antes de producción
El environment production en GitHub tiene configurados Required Reviewers: personas designadas que deben aprobar el deploy antes de que continúe. Sin su aprobación, el job queda esperando hasta 30 días y luego expira. Esto es el equivalente directo a la FSR del SDL de Microsoft.
Configuración en GitHub — Settings → Environments → production
MANUAL 
# En GitHub: Settings → Environments → production
# ✅ Required reviewers: [security-team, tech-lead]
# ✅ Wait timer: 0 minutes
# ✅ Deployment branches: main only

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production              # ← requiere aprobación manual
      url: https://mi-app.com
    steps:
      - name: Deploy a producción
        run: |
          kubectl set image deployment/mi-app \
            mi-app=ghcr.io/tu-org/mi-app:${{ github.sha }}

      - name: Verificar headers de seguridad HTTP
